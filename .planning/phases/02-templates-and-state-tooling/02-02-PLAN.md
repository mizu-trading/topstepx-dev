**api-integration-plan.md (TPL-03):**
Template for API integration planning documents. Sections: Authentication (login method, token storage, refresh strategy), REST Endpoints Used (table: endpoint, purpose, frequency), WebSocket Subscriptions (table: hub, method, events handled), Error Handling (rate limit strategy, order rejection handling, connection recovery), Environment Variables (table: var name, source, required). Include example for a basic trading bot connecting to TopStepX with market data streaming and order placement.

**risk-parameters.md (TPL-04):**
Template for risk parameter documents that capture position sizing, loss limits, and safety constraints. Sections: Account Context (account type, starting balance, platform rules), Position Sizing (max contracts, scaling rules), Loss Limits (max loss per trade, max daily loss, max drawdown, action on breach), Order Defaults (default order type, bracket orders config with stop-loss ticks and take-profit ticks), Safety Overrides (kill switch conditions, auto-flatten rules). Reference SAF-01 from safety-patterns.md. Include example for a conservative ES futures setup.

**bot-scaffold-js.md (TPL-05 - JavaScript/TypeScript):**
Template for JS/TS bot starter code. This is a NEAR-RUNNABLE scaffold, not a placeholder-heavy template. The code must include ALL of the following as non-optional defaults (copy exact patterns from topstepx/references/safety-patterns.md):

1. **Enum constants** (SAF-01): OrderSide, OrderType, OrderStatus, PositionType with all named values -- NEVER bare integers
2. **TokenManager class** (SAF-02): Constructor with initial token, 23-hour proactive refresh timer, validate() method calling /api/Auth/validate, getToken() accessor
3. **RateLimiter class** (SAF-03): RATE_LIMITS constant (HISTORY: 50/30s, GENERAL: 200/60s), checkLimit() with sliding window
4. **createOrder function** (SAF-01): Takes side, size, contractId, accountId. Applies Math.min(size, maxSize) for position sizing. ALWAYS includes stopLossBracket and takeProfitBracket with configurable ticks
5. **Bot class structure**: constructor (config with API credentials, risk params), connect() (auth + WebSocket setup), onQuote() placeholder, evaluateSignal() placeholder, placeOrder() using createOrder with brackets, shutdown() for cleanup
6. **Error handling** (SAF-05): try/catch on all API calls, check response.success, handle rejected orders, WebSocket reconnection with re-subscribe

Mark strategy-specific sections (indicator calculation, entry/exit logic, signal evaluation) with clear `// --- YOUR STRATEGY LOGIC ---` placeholder markers. Everything else (safety, auth, connection, order placement) is implemented and non-negotiable.

Include a guidelines section explaining that agents fill in strategy placeholders but must NOT modify safety infrastructure code.

**bot-scaffold-python.md (TPL-05 - Python):**
Same structure as JS scaffold but in Python. Use equivalent patterns:
1. Enum constants as class attributes or IntEnum
2. TokenManager class with asyncio timer for 23hr refresh
3. RateLimiter with sliding window (collections.deque + time)
4. create_order() function with bracket defaults
5. Bot class with async connect(), on_quote(), evaluate_signal(), place_order(), shutdown()
6. Error handling with try/except, response checking, reconnection

Use `aiohttp` for REST calls and `pysignalr` for WebSocket (per PINESCRIPT.md library mapping). Mark strategy placeholders same as JS version.

**pinescript-conversion.md (TPL-06):**
Template for PineScript conversion report documents. Sections: Source Analysis (PineScript version, strategy type, indicators used, entry/exit logic summary), Repainting Audit (table of indicators checked, repainting risk level, mitigation applied -- reference SAF-04 from safety-patterns.md), Multi-Timeframe Audit (request.security usage, lookahead bias check, offset applied), Conversion Mapping (table: PineScript construct -> TopStepX equivalent), Bar Close Policy (execution model decision: confirmed-bar vs tick-based, with rationale), Safety Integration (which SAF-01..05 patterns applied), Generated Files (list of output files). Include example for converting a simple RSI overbought/oversold strategy.

Include example for JS-to-Python adaptation of a simple trading bot.
